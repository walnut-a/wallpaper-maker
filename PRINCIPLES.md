# PRINCIPLES

这个文件用于约束同类项目的产品与实现方向。

它不替代 README。
- `README` 负责说明项目做什么、怎么用、怎么部署。
- `PRINCIPLES` 负责说明这个项目为什么这样设计，以及后续新增功能时应遵守什么边界。

适用范围：
- 单文件前端工具
- 可直接托管在 GitHub Pages
- 不依赖后端即可完成核心功能
- 面向轻量创作、整理、生成、导出类体验

## 1. 最高优先级原则

### 1.1 单文件优先
项目应尽量收敛为一个可直接打开和托管的 `HTML` 文件。

判断标准：
- 样式、结构、脚本尽量内聚
- 不为了“工程化”引入不必要的构建链
- 新增能力前先判断是否会破坏单文件交付

### 1.2 静态托管完整可用
项目部署到 GitHub Pages 这类纯静态环境后，核心功能必须完整可用。

判断标准：
- 核心链路不能依赖后端
- 不能要求本地服务、数据库、鉴权回调或特殊运行环境
- 用户应能在静态页面里完成输入、处理、预览、导出

### 1.3 打开即用
用户不应先理解工程结构，才能开始使用产品。

判断标准：
- 页面打开后立即进入可操作状态
- 默认状态本身就是可理解的
- 不把“安装、配置、启动”变成使用前提

## 2. 产品原则

### 2.1 功能闭环要短
界面应尽快把用户带到核心动作，不绕路。

判断标准：
- 首屏尽量直接进入主要任务
- 核心输入、核心结果、核心导出之间路径短
- 少做与目标无关的说明和铺垫

### 2.2 本地优先
能在浏览器本地完成的能力，不交给服务端。

判断标准：
- 生成、预览、导出等优先使用浏览器原生能力
- 外部依赖只能作为补充，不能成为主流程唯一前提

### 2.3 默认状态就是产品态度
默认文案、默认参数、默认样式，应该代表项目想传达的质量。

判断标准：
- 空状态不应像未完成页面
- 示例内容应能准确展示项目能力
- 默认配置应适合大多数使用场景

## 3. 设计原则

### 3.1 内容优先
界面服务内容，不反过来让内容服务界面。

判断标准：
- 用户最终关心的结果应是视觉中心
- 编辑器、容器、背景都应该退后
- 装饰不能盖过核心输出

### 3.2 容器退后
工作台可以精致，但不能比作品本身更强势。

判断标准：
- 外层容器的圆角、阴影、边框、颜色要比结果层克制
- 避免出现“编辑器比作品更像作品”的情况

### 3.3 一致性优先
同一层级的模块要共享同一套节奏。

判断标准：
- 内边距一致
- 标题起点一致
- 圆角体系一致
- 辅助文案密度一致

### 3.4 少解释，多自解释
优先通过结构、顺序和命名传达含义，而不是堆文案提示。

判断标准：
- 删除某段说明后，如果用户仍能顺利完成任务，就应该删
- 标签文本要清楚，但提示块要克制

### 3.5 移动端单独判断
移动端不是桌面版压缩。

判断标准：
- 模块顺序按真实使用流程组织
- 标题和说明的宽度、字号、换行要单独调
- 不能把桌面层级原样缩放到手机上

## 4. 工程原则

### 4.1 依赖节制
每一个外部依赖都必须有明确理由。

判断标准：
- 是否真的提升核心体验
- 是否破坏单文件交付
- 是否影响 GitHub Pages 可用性
- 是否存在浏览器原生替代方案

### 4.2 降级明确
如果某项能力依赖外部资源，失败时必须可理解、可接受。

判断标准：
- 用户知道什么失效了
- 主流程尽量仍可继续
- 不出现静默失败

### 4.3 可复制、可迁移、可维护
这个项目类型的价值之一就是容易分发和复用。

判断标准：
- 新项目可以直接复制基础结构
- 托管方式简单
- 文件结构和逻辑足够容易理解

## 5. 决策检查清单

每次新增功能或改界面前，先问这几个问题：

1. 这次改动会不会破坏单文件交付？
2. 这次改动在 GitHub Pages 上能否完整成立？
3. 这次改动是在强化核心结果，还是在增加界面存在感？
4. 这段文案删掉后，用户还能不能完成任务？
5. 外层容器是不是比内容本身更抢眼？
6. 移动端是否仍然符合真实使用顺序？

## 6. 一句话总结

这类项目的目标不是做一个“工程很完整”的网页，而是做一个：

`单文件交付、静态环境可用、打开即用、内容优先、结果导向` 的轻量工具。
